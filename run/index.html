NATS JetStream vs Redis — Throughput Benchmark

Single-machine, rootless Podman, 16GB memory cap per server, systems run sequentially (never concurrently) to avoid CPU/cache/bandwidth contention. 90,000 messages per run, 4 message sizes × 3 concurrency levels × 5 repeats, across 3 durability tiers.
⚠
Read the numbers below with these caveats in mind:
  • NATS p50/p99 latency is only populated at concurrency=1. run.sh's parse_nats_output reads percentiles off the last line matching msgs/sec — for concurrency>1 that's the "aggregated stats" line, which doesn't carry percentiles (those are on the following "latencies per operation" line). Multi-client NATS runs recorded throughput correctly but left latency columns blank — a parsing gap worth fixing in run.sh, not a server issue.
  • redis-benchmark's latency values are coarsely quantized at low concurrency — most concurrency=1 runs report exactly 0.015ms for p50/p95/p99 alike, a histogram-bucket artifact of the tool itself, not a real flat latency distribution.
  • Tier1 throughput compares different client concurrency models, not just the two servers: nats bench pub is fully async/non-blocking client-side, while redis-benchmark blocks for each reply before sending the next command. The ~35× tier1 gap partly reflects that difference, not purely server-side speed.
